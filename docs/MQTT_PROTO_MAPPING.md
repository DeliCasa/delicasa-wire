# MQTT-to-Proto Mapping: Capture Protocol

**Package**: `@delicasa/wire@0.4.0`
**Scope**: Maps the 4-phase MQTT capture protocol between ESP32 and PiOrchestrator to the `CaptureImageResponse` proto message.

## Architecture Context

```
ESP32 (MQTT) → PiOrchestrator (translates) → BridgeServer (Connect RPC)
                    ↑                              ↑
              Receives MQTT messages          Receives CaptureImageResponse
              Assembles proto response        via CaptureService.CaptureImage
```

The PiOrchestrator is the **translation boundary**. It receives 4 MQTT message phases from the ESP32, assembles the image data, optionally uploads to R2, and returns a `CaptureImageResponse` proto message via Connect RPC.

**Non-negotiable**: ESP32 stays on MQTT. See [ADR-001](./ADR-001-ESP32-STAYS-MQTT.md) for rationale.

## 4-Phase MQTT Capture Protocol

### Phase 1: Ack

**MQTT Topic**: `camera/response/{request_id}/ack`
**Zod Schema**: `MqttCaptureAckSchema` (`src/zod/mqtt-capture.ts`)
**Direction**: ESP32 → PiOrchestrator

| MQTT Field | Type | Proto Mapping |
|-----------|------|---------------|
| `request_id` | string | Used internally for correlation — not in proto response |
| `device_id` | string | → `CaptureImageResponse.camera_id` |
| `action` | `"capture"` | N/A (protocol control) |
| `status` | `"in_progress"` | N/A (protocol control) |
| `timestamp` | ISO 8601 | N/A (not the capture timestamp) |
| `correlation_id` | string? | → `CaptureImageResponse.correlation_id` (if present) |
| `session_id` | string? | Used for evidence tracking, not in capture response |

**PiOrchestrator action**: Record that the camera acknowledged the capture command. Start timeout timer.

### Phase 2: Info

**MQTT Topic**: `camera/response/{request_id}/info`
**Zod Schema**: `MqttCaptureInfoSchema` (`src/zod/mqtt-capture.ts`)
**Direction**: ESP32 → PiOrchestrator

| MQTT Field | Type | Proto Mapping |
|-----------|------|---------------|
| `request_id` | string | Internal correlation |
| `device_id` | string | → `CaptureImageResponse.camera_id` |
| `success` | boolean | If false → `CaptureStatus.CAPTURE_STATUS_FAILED` |
| `image_size` | int | → `CaptureImageResponse.image_size_bytes` |
| `content_type` | string | → `CaptureImageResponse.content_type` |
| `total_chunks` | int | Used to validate chunk reassembly |
| `timestamp` | ISO 8601 | → `CaptureImageResponse.captured_at` |
| `timings.capture_ms` | int | Diagnostic only — not in proto response |
| `timings.validate_ms` | int | Diagnostic only |
| `timings.encode_ms` | int | Diagnostic only |

**PiOrchestrator action**: Extract image metadata. Prepare buffer for `total_chunks` chunks. Set `captured_at` from this timestamp.

### Phase 3: Chunk (repeated)

**MQTT Topic**: `camera/response/{request_id}/chunk`
**Zod Schema**: `MqttCaptureChunkSchema` (`src/zod/mqtt-capture.ts`)
**Direction**: ESP32 → PiOrchestrator (sent `total_chunks` times)

| MQTT Field | Type | Proto Mapping |
|-----------|------|---------------|
| `request_id` | string | Internal correlation |
| `chunk_index` | int | Used for ordered reassembly |
| `chunk_data` | base64 string | Reassembled → `CaptureImageResponse.image_data` (if no R2 upload) |

**PiOrchestrator action**: Append chunk to image buffer at `chunk_index` position. When all chunks received, reassemble full JPEG.

### Phase 4: Complete

**MQTT Topic**: `camera/response/{request_id}/complete`
**Zod Schema**: `MqttCaptureCompleteSchema` (`src/zod/mqtt-capture.ts`)
**Direction**: ESP32 → PiOrchestrator

| MQTT Field | Type | Proto Mapping |
|-----------|------|---------------|
| `request_id` | string | Internal correlation |
| `success` | boolean | Determines `UploadStatus` (see error mapping below) |
| `chunks_sent` | int | Validation — must match `total_chunks` from Info |
| `device_id` | string | → `CaptureImageResponse.camera_id` |
| `total_chunks` | int | Cross-check with Info phase |
| `timestamp` | ISO 8601 | End-of-capture timestamp (diagnostic) |
| `timings.total_ms` | int | Diagnostic only |

**PiOrchestrator action**: Validate all chunks received. If R2 configured, upload reassembled image → set `object_key` and `upload_status = UPLOAD_STATUS_UPLOADED`. If no R2, set `image_data` with raw bytes and `upload_status = UPLOAD_STATUS_UNVERIFIED`. Generate `evidence_id`. Return `CaptureImageResponse`.

## Proto Response Assembly

After all 4 phases complete, PiOrchestrator constructs:

```protobuf
message CaptureImageResponse {
  correlation_id  ← from Ack.correlation_id or request context
  evidence_id     ← generated by PiOrchestrator
  camera_id       ← from Ack.device_id / Info.device_id
  captured_at     ← from Info.timestamp
  content_type    ← from Info.content_type
  image_size_bytes ← from Info.image_size
  object_key      ← from R2 upload result (if uploaded)
  image_data      ← from reassembled chunks (if no R2)
  upload_status   ← UPLOADED (R2 success) / FAILED (R2 error) / UNVERIFIED (no R2)
  cached          ← false (first capture) / true (idempotency hit)
}
```

## Error Scenario Mapping

| Scenario | MQTT Signal | Proto CaptureStatus | Proto UploadStatus | Notes |
|----------|------------|--------------------|--------------------|-------|
| Successful capture + R2 upload | Complete.success=true | CAPTURE_STATUS_CAPTURED | UPLOAD_STATUS_UPLOADED | Happy path |
| Successful capture, no R2 | Complete.success=true | CAPTURE_STATUS_CAPTURED | UPLOAD_STATUS_UNVERIFIED | image_data populated instead |
| Capture failed on device | Info.success=false | CAPTURE_STATUS_FAILED | N/A | No image data available |
| Timeout (no Ack received) | No MQTT response | CAPTURE_STATUS_TIMEOUT | N/A | PiOrchestrator timeout timer fires |
| Timeout (Ack but no Complete) | Ack received, no Complete | CAPTURE_STATUS_TIMEOUT | N/A | Partial protocol — device hung |
| Chunk mismatch | chunks_sent != total_chunks | CAPTURE_STATUS_FAILED | UPLOAD_STATUS_FAILED | Data corruption |
| R2 upload failure | Complete.success=true | CAPTURE_STATUS_CAPTURED | UPLOAD_STATUS_FAILED | Image captured but storage failed |

## MQTT Topic Patterns

All topics are constructed via `MqttTopics` helpers in `src/zod/mqtt-topics.ts`:

| Purpose | Topic Pattern | Helper |
|---------|--------------|--------|
| Send capture command | `camera/command/{device_id}` | `MqttTopics.cameraCommand(deviceId)` |
| Receive response | `camera/response/{request_id}` | `MqttTopics.cameraResponse(requestId)` |
| Receive ack | `camera/response/{request_id}/ack` | `MqttTopics.cameraResponse(requestId, "ack")` |
| Receive info | `camera/response/{request_id}/info` | `MqttTopics.cameraResponse(requestId, "info")` |
| Receive chunk | `camera/response/{request_id}/chunk` | `MqttTopics.cameraResponse(requestId, "chunk")` |
| Receive complete | `camera/response/{request_id}/complete` | `MqttTopics.cameraResponse(requestId, "complete")` |
| Camera status | `camera/status/{device_id}` | `MqttTopics.cameraStatus(deviceId)` |
