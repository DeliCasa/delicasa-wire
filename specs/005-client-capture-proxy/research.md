# Research: Client-Facing Capture Proxy Service

**Feature**: 005-client-capture-proxy
**Date**: 2026-03-06

## Decision 1: Cross-Package Enum Import vs Redeclare

**Decision**: Use `string capture_tag` in the client-facing proto (not the `CaptureTag` enum from `delicasa.device.v1`).

**Rationale**: The existing `delicasa.v1` protos never import from `delicasa.device.v1`. The `Image` message in `image.proto` already uses `string capture_tag` (line 21) rather than the device-layer `CaptureTag` enum. This is an intentional boundary separation — client-facing protos should not depend on device-layer types. The BridgeServer is responsible for mapping between the string value and the device-layer enum when proxying.

**Alternatives considered**:
- Import `CaptureTag` enum from `delicasa.device.v1.evidence.proto` — rejected because it creates a cross-package dependency, violates the boundary separation principle, and makes client-facing types depend on device-layer evolution.
- Redeclare a `CaptureTag` enum in `delicasa.v1` — rejected because it creates two sources of truth for the same enum values and introduces sync burden.

## Decision 2: CaptureRequestStatus Enum Design

**Decision**: Define a `CaptureRequestStatus` enum in the client-facing capture service proto with values: `UNSPECIFIED`, `ACCEPTED`, `COMPLETED`, `FAILED`, `TIMEOUT`.

**Rationale**: This represents the client's view of the capture lifecycle — distinct from the device-layer `CaptureStatus` which tracks the MQTT protocol state. The client doesn't need to know about device-level statuses like `PENDING` (which maps to `ACCEPTED` from the client's perspective). `ACCEPTED` means BridgeServer has forwarded the request; `COMPLETED` means evidence is available.

**Alternatives considered**:
- Reuse `delicasa.device.v1.CaptureStatus` — rejected for same cross-import reasons as Decision 1.
- Use a simple boolean `success` field — rejected because it can't represent the in-progress state needed for async capture polling.

## Decision 3: UploadStatus in Client Response

**Decision**: Import `UploadStatus` from within `delicasa.v1` (define it there) rather than referencing device-layer `UploadStatus`.

**Rationale**: The client-facing response needs to indicate whether the image was uploaded to R2. However, `UploadStatus` currently lives in `delicasa.device.v1.capture_service.proto`. Since we can't cross-import, we have two options: (a) redeclare in `delicasa.v1` or (b) use a string field. Given that `UploadStatus` has only 4 values and is stable, redeclaring is acceptable. We'll put it in a shared `capture.proto` entity file within `delicasa.v1`.

**Alternatives considered**:
- String field for upload status — rejected because it loses type safety and enum validation.
- Skip upload status in client response entirely — rejected because NextClient needs to know if the image URL is available.

## Decision 4: Service Naming

**Decision**: Name the service `CaptureService` under `delicasa.v1` (not `DeviceProxyService`).

**Rationale**: From the NextClient developer's perspective, they're "capturing an image" — the proxy nature is an implementation detail. The service name should reflect the domain action, not the infrastructure pattern. The service boundary documentation will make the routing clear.

**Alternatives considered**:
- `DeviceProxyService` — rejected as it exposes internal architecture to consumers and creates confusion if the proxy layer is ever removed.
- `ImageCaptureService` — rejected for consistency with the device-layer `CaptureService` naming.

## Decision 5: Version Bump Strategy

**Decision**: Minor bump to v0.5.0.

**Rationale**: Adding a new service is a feature addition, not a patch fix. Consistent with v0.3.0 (added device services) and v0.4.0 (added testing exports). No breaking changes to existing services.

## Decision 6: GetCaptureStatus Lookup Fields

**Decision**: `GetCaptureStatus` accepts `request_id` as the primary lookup key. `correlation_id` is included for tracing but is not a lookup field.

**Rationale**: The `request_id` is generated by BridgeServer when it accepts a `RequestCapture` call and is guaranteed unique. `correlation_id` is client-supplied and may not be unique across requests. Using `request_id` as the lookup key is simpler and unambiguous. The spec mentioned "either `request_id` or `correlation_id`" but using a single key simplifies the implementation.
